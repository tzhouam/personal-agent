# A.8 accountId 验证指南（多用户上线前的硬性关卡）

## 一、这是在验证什么

多用户（`multi_tenant`）模式的整套设计都建立在一个假设上：**每个微信账号在
`before_dispatch` 时刻，OpenClaw 会把一个稳定、且各账号互不相同的 `accountId`
交给桥接插件**（还要有 `channel` 和可用的 `sessionKey`）。桥接就是靠这个
`accountId` 把消息路由到对应用户（`accountId → uid`）。

这个假设**只能在你的真机上、用两个微信账号验证**，我在会话里无法代跑。验证通过
之前，**不要**把生产环境切到 `DEPLOYMENT_MODE=multi_tenant`。详见
`../doc/DESIGN_MULTI_USER.md` §A.8。

本指南使用一个**只读探针插件**（`openclaw-plugin-spike/`）：它只打印日志、永远
返回 `undefined`，**不会拦截任何消息**，所以可以和你正在用的
`personal-agent-bridge` 同时开着，互不影响。

---

## 二、前置条件

- 一台已经跑着 OpenClaw 网关、且 `personal-agent-bridge` 正常工作的机器。
- **两个微信账号**，都登录到**同一个** OpenClaw 网关。
- 能看到网关的控制台输出 / 日志（就是平时能看到 `[personal-agent-bridge]` 那些行
  的地方）。

---

## 三、步骤

### 步骤 1：安装探针插件

和安装主桥接插件同样的方式，用 `--link` 指向仓库里的探针目录：

```bash
openclaw plugins install --link <仓库路径>/personal-agent/openclaw-plugin-spike
```

> `--link` 会保持指向仓库，之后 `git pull` 即可更新，无需重装。

### 步骤 2：在 `openclaw.json` 里启用它

在网关的 `openclaw.json`（通常在 `~/.openclaw/openclaw.json`）的
`plugins.entries` 下，为探针加上这两项：

| 配置项 | 值 | 说明 |
| --- | --- | --- |
| `plugins.entries.a8-spike-probe.enabled` | `true` | 启用探针 |
| `plugins.entries.a8-spike-probe.hooks.allowConversationAccess` | `true` | **必须**——非内置插件不给这个权限就注册不上会话钩子，且是静默失败 |

例如：

```json
{
  "plugins": {
    "entries": {
      "personal-agent-bridge": { "enabled": true, "hooks": { "allowConversationAccess": true } },
      "a8-spike-probe":        { "enabled": true, "hooks": { "allowConversationAccess": true } }
    }
  }
}
```

**保留** `personal-agent-bridge` 开着——探针只观察，不抢消息。

### 步骤 3：重启网关并确认探针已加载

重启网关后，确认探针处于 loaded 状态：

```bash
openclaw plugins inspect a8-spike-probe --runtime
```

应能看到它注册了 `before_dispatch` 与 `message_received` 钩子。

### 步骤 4：用两个账号分别发消息

- 用**账号 A** 发两条微信消息（例如先发 `hi`，再发 `/status`）。
- 用**账号 B** 也发两条。

（发两条是为了验证同一账号的 `accountId` 是否**稳定不变**。）

### 步骤 5：查看日志

在网关日志里筛选探针输出：

```bash
tail -f <网关日志文件> | grep A8-SPIKE
```

每条消息会打印一对行，形如：

```
[A8-SPIKE] before_dispatch ctx={"channel":"weixin","accountId":"wx-acctA","sessionKey":"s1","_keys":[...]}
[A8-SPIKE] before_dispatch event={"body":"hi","content":"hi","cleanedBody":"","_keys":[...]}
```

---

> **✅ 2026-07-16 已在本部署验证通过**：`ctx.accountId = 4fbce27f431d-im-bot`
> 在 `before_dispatch` 存在；正文在 `event.body`/`event.content`。**实测修正**：
> channel 字段在 **`event.channel`** 上，值是插件全名 **`"openclaw-weixin"`**
> （`ctx` 上只有 `channelId`，没有 `ctx.channel`）；`sessionKey` 是账号级的
> `agent:main:main`，按会话记忆要用 `ctx.conversationId`。桥接代码已按实测修正。

## 四、判定清单（逐条对照 `before_dispatch` 的行）

- [ ] **channel 可识别** —— `event.channel`（或 `ctx.channelId`）为
      `"openclaw-weixin"`（或 `"weixin"`）。
- [ ] **`ctx.accountId` 存在且非空。**
- [ ] **稳定**：账号 A 的两条消息里 `accountId` **相同**；账号 B 的两条也相同。
- [ ] **互不相同**：A 的 `accountId` ≠ B 的 `accountId`。
- [ ] **正文可读**：文本在 `event.body` 或 `event.content` 里（注意
      `event.cleanedBody` 这里预期是**空**的——这正是桥接从
      `before_agent_reply` 迁到 `before_dispatch` 的原因）。
- [ ] **`ctx.sessionKey`** 存在，且同一会话内稳定。

**全部打勾 = 关卡通过。**

---

## 五、结果处理

### ✅ 全部通过

1. 为每个用户绑定其真实 `accountId`：
   ```bash
   assistant admin add-user <uid> --display "名字"
   assistant admin bind-channel <uid> weixin <accountId>   # 用日志里读到的真实值
   ```
2. 设置桥接令牌（`.env` 里的 `SERVE_TOKEN` 与下面这条用**同一个值**）：
   ```bash
   assistant admin set-bridge-token <token>
   ```
3. **为每个账号配置 OpenClaw 的发信人白名单/配对，并实际验证**：用一个不在
   白名单里的微信号给账号 A 发消息，必须被拒绝而不是被回答。
   > `accountId` 只做**租户路由**（这条消息属于谁的租户），**不做发信人鉴权**
   > （是谁发的）——鉴权靠每个账号自己的白名单/配对（设计文档 §11.3）。
4. 在 `.env` 里切换：`DEPLOYMENT_MODE=multi_tenant`，重启 `assistant serve`。

### ❌ `accountId` 缺失或不稳定

不要切换模式。看探针打印的 **`_keys`** 列表（`ctx` / `event` 上的**完整字段名**），
它会告诉我们微信通道到底提供了什么——`accountId` 也许换了个名字，或者挂在
`event` 上。把那几行 `[A8-SPIKE]` 日志发我，我据此调整
`openclaw-plugin/index.js` 里的 `ctx?.accountId` 读取逻辑，然后再启用。

---

## 六、验证完成后清理

在 `openclaw.json` 里把 `a8-spike-probe` 删掉（或 `enabled: false`），重启网关。
探针不写任何文件、不留任何状态，删掉即干净。
```bash
openclaw plugins uninstall a8-spike-probe   # 或者只在 openclaw.json 里禁用
```
